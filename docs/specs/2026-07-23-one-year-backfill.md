## 1. 关键结论：一年可回填的只有「一手公告」，媒体不行

6 个源分成两类形态，能不能回填一年，取决于源有没有「按公司 + 按时间区间」的查询能力。

### ✅ 可全年回填：一手公告（PRIMARY）

**东方财富公告接口**（实测最优）：

```
np-anotice-stock.eastmoney.com/api/security/ann
  ?stock_list=<code>&begin_time=2025-07-23&end_time=2026-07-23
  &page_size=50&page_index=N&ann_type=A&sr=-1
```

实测结论：
- 支持**按股票 + 按日期区间**精确查询，翻页到空为止，`total_hits` 与巨潮交叉验证一致（宁德 226 vs 222，华阳 96 vs 96）；
- `page_size=50` 有效，**巨潮硬上限只有 30**（和之前 `pz=200→100` 是同一类服务端截断）；
- 不需要巨潮那样先查 `orgId`，**每家省一个请求**；
- 返回 `art_code`，可复用已有 `enrich.ts` 拿正文。

→ **回填主力选东财，不选巨潮**。平均 3.5 页/家，801 家共约 **2,800 个请求**。

### ❌ 不能全年回填：媒体（MEDIA）

| 源 | 实测结果 | 结论 |
|---|---|---|
| 东财个股资讯（统一搜索） | `pageIndex=20`（第 400 条）仍停在昨天。它是相关性搜索不是编年存档，翻不回一年 | 不可行 |
| 新浪个股新闻存档 | 可翻页，但**约 28 页封顶**（~1,100 条）：中盘股只到 2025-10，龙头股只到 2 个月前；且列表**只有标题+链接、没有摘要正文** | 代价 2.2 万请求换标题党卡片，不值 |
| 东财快讯 / 华尔街见闻 | 全市场 7×24 流，无按公司历史接口。按当前速率一年约 **20 万条**宏观碎讯，多数 0 绑定或只绑板块 | 会把 12 轮降噪成果全部推翻 |

**所以「过去一年的新闻」现实的落点是：一年的一手公告 + 保持媒体的滚动窗口。** 这恰好和解牛「一手」的定位一致——投资者时间线上真正该留下的是公告，不是媒体转述。

## 2. 三个必须先修的地雷（不修，回填会把产品搞坏）

### 地雷 ①：通知中心会被一年前的新闻淹没

`notifications.list` 按 `createdAt desc` 排序，`unreadCount` 按 `createdAt > seenAt` 计数。回填行的 `createdAt` 全是「现在」→ **一年前的公告会全部变成「未读新动态」**，未读数瞬间上万。

已核查：其余所有 feed（news / entity / portfolio / feed / ask / user-thesis 共 28 处）都用 `publishedAt`，**只有 notifications 用 createdAt**，改动面很小。

修法：`notifications` 的排序与未读水位线改用 `publishedAt`，并加「只有 `publishedAt ≥ now-7d` 才算通知」的新鲜度窗口。

### 地雷 ②：去重窗口按 createdAt 只看 7 天 → 回填必产生重复

`runner.ts` 载入去重集合的条件是 `createdAt >= now-7d`。实测**已有 3,360 条巨潮 + 600 条东财公告的 createdAt 已超过 7 天**，回填时看不见它们。而巨潮的 `externalId` 是 `adjunctUrl`、东财是 `art_code`，hash 不同 → 重叠期（2026-03-13 至今）的同一份公告会**被存成两条**。

这和上一轮「跨源重复」是同一类问题，必须在回填路径单独处理：按**本次回填的 publishedAt 区间**载入该公司已有条目，用 `(实体 + 归一化标题)` 判重，不用 7 天 createdAt 窗口。

### 地雷 ③：一年前的例行公告没有价值，全量入库＝糊墙

对 240 条真实公告样本跑现有过滤器：

- 过现有标题过滤器：200/240（83%）
- 其中**能识别出事件类型的只有 26 条（11%）**，其余 174 条是：董事会决议公告、审计委员会履职报告、独立董事述职报告、股东会通知/决议/资料、法律意见书、ESG 报告、薪酬方案、续聘会计师……

实时 ingest 收这些还说得过去（当天的公告有时效价值）；**一年前的治理文书没有任何回看价值**，10 万条入库只会让个股页变成一堵墙。

修法：回填走一道比实时**更严的材料门槛**——只留事后回看仍算「事件」的：经营（重大合同/中标/投产/产能）、资本（收购/重组/增发/可转债/回购）、股权（增减持/解禁/质押/激励/控制权）、业绩（定期报告/预告/快报/分红）、监管（问询/立案/处罚/停复牌/异常波动）。

原型门槛实测存活 **55/240 = 22.9%**，抽查被丢的样本确认没有明显误杀（丢的是股东会决议、法律意见书、董事会工作报告、薪酬方案）。补上「质押/解质押、H股发行备案、试运转/达产」后预计 **≈25%**。

→ 801 × 128 × 25% ≈ **2.6 万条材料级公告，约 32 条/家/年**，即平均每 11 天一个真事件。合理。

## 3. 实施方案

### Phase 0 — 拆雷（先做，约半天）
1. `notifications` 排序 / 未读水位线 `createdAt` → `publishedAt` + 7 天新鲜度窗口；补测试。
2. `runner.ts` 增加可注入的去重窗口（回填模式按 publishedAt 区间取），保持实时路径行为不变；补测试。
3. `src/lib/relevance.ts` 增加 `isMaterialFiling()`（回填专用材料门槛）+ 单测。

### Phase 1 — 一年公告回填（主体）
- 新源：`eastmoneyAnnForCode(code, from, to)`，翻页至空。
- 新脚本：`src/scripts/backfill-year.ts --months=12 --limit=N`
  - 按「当前绑定资讯最少」优先（自愈，先补最空的）；
  - 分批可续跑，hash 幂等，重跑安全；
  - 每家之间限速 400ms，失败跳过不中断。
- 分 ~8 批后台跑完，全程 **0 token**（已核查 `runner.ts` 全链路纯规则、无 AI 调用），只花时间和存储。
- 预期：DB 60 MB → 约 110 MB；个股页从 11 条/3 周 → **约 43 条/一年**。

### Phase 2 — 让这一年在界面上真的看得见
回填了没地方显示等于白填。个股页现有查询大多是 `publishedAt >= 7/30 天`：
- 个股页新增「大事记」：按季度/月折叠的一年材料事件时间线；
- 定期报告（年报/中报/季报）自动作为时间轴锚点；
- `event-timeline` / `logic-tracker` 的回看窗口从 30 天放宽到可选一年。

### Phase 3 — 媒体侧（有限度，不追一年）
- 不做媒体全年回填（理由见 §1）。
- 把现有轮转刷新的每股媒体深度从 `pageSize=5` 提到 20（成本几乎不变）。
- 对 0 条的 6 家 + <5 条的 37 家做定向修补，属于修 bug 不是铺量。

### Phase 4 — 验证（每阶段都做，不自我report成功）
- 回填前后各跑一次 `coverage-report.ts` + `effective-coverage.ts` 存档对比；
- 抽 5 家公司页 PC + 移动截图人眼验收；
- 专项确认：**通知中心未读数没有爆炸**、**同公司无重复标题**；
- `npm test` 全绿 + `build` exit 0（先构建成功再重启，分步执行）。

## 4. sway 的拍板（2026-07-23）

> 「做，条数多。看看有没有一些数据源来获取到最近一年的数据。」

1. **材料门槛 → 取宽档（83%）**。不加回填专用材料门槛，只走现有的标题过滤器；
   例行治理公告照收。为避免「条数多 = 糊墙」，改用**分层**而非过滤：
   例行公告 importance=45、重磅事件 ≥55，个股页新增「大事记」tab 只收 ≥55 并按月折叠，
   完整清单留在「公告」tab。要多有多、要看脉络也看得清。
2. **继续挖一年期数据源** → 见 §4.1。

### 4.1 第二轮源勘察（全部实测，2026-07-23）

| 源 | 能否按公司拉一年 | 结论 |
|---|---|---|
| **东财·券商研报** `reportapi.eastmoney.com/report/list` | ✅ 支持 `code` + `beginTime/endTime` + 翻页 | **采用**。茅台 72 篇/年、宁德 38、汇川 21、罗博特科 2，平均约 15 篇/只，多数一页取完，801 只约 900 个请求 |
| 巨潮·互动易 `irm.cninfo.com.cn` | ❓ 接口返回 200 但 `totalRecord=0`，三种参数组合都试过；备选端点 405 / 404 | **放弃**（参数需从站点 JS 逆推，不值当；没拿到就是没拿到） |
| 雪球 `stock_timeline` | ❌ 阿里云 WAF 拦截 | 放弃 |
| 同花顺 F10 新闻 | ❌ 空响应 | 放弃 |
| 东财·机构调研 datacenter | ⚠️ 可查但量极小（宁德一年仅 4 条） | 不单独接 |

**研报的合规处理（铁律②不荐股/不喊价）**：接口同时返回 `emRatingName`(买入/增持)、
`indvAimPriceT/L`(目标价)、`predict*Eps/Pe`(盈利预测)。这些字段**在类型定义里就不声明**，
从源头杜绝误用；标题命中评级/目标价语言的整条丢弃（`lib/compliance.isRatingHeadline`）。
解牛只记录「哪家机构在哪天发了一篇什么主题的研报」这一**事件**，不记录也不展示它的结论。

**研报的归属处理**：机构名放**标题后缀**「…（国信证券）」而不是前缀——前缀会被
`stripEntityPrefix` 当作「公司名:」剪掉，导致不同券商同日的同名研报（「2026年半年报点评」）
被判成重复而丢失。相应地 runner 新增 `isReportPublisherSuffix` 把发布机构自身的绑定剪掉，
否则券商 feed 会被自家研报淹没（这是 run3 修过的旧疾）。

### 4.2 施工中发现的第四颗雷：按重要性排序的流没有时间窗

`news.important`（首页重大动态）、`feed.myFeed`（自选流）、`entity.ecosystem`（板块动态）
都是 `orderBy: [importance desc, publishedAt desc]` 且**不带时间窗**。

回填前库里只有约 3 周数据，「近期」是被数据范围**隐含**保证的；回填一年后这个隐含约束消失，
一条一年前的停牌(90 分)会永久钉在首页第一条。已加 `SURFACING_WINDOW_DAYS=30` 显式窗口——
这不是改行为，是把回填前就存在的行为写出来。

## 5. 一句话总结

一年的**一手公告**能干净地拿到（2,800 个请求、0 token、约 2.6 万条材料事件，个股页从 3 周变一年）；一年的**媒体新闻**拿不到也不该拿。先拆通知/去重/材料门槛三颗雷，再回填，再把界面打开到一年——顺序错了会把已经治好的噪声问题重新引爆。
